## 二、Topic 规则

所有 Topic 都以 `longarch/device/` 开头，后端用通配符订阅：

| Topic 模式 | 方向 | 用途 |
|-----------|------|------|
| `longarch/device/telemetry/{deviceNo}` | **设备 → 后端** | 传感器数据上报 |
| `longarch/device/heartbeat/{nodeNo}` | **设备 → 后端** | 边缘节点心跳 / 在线状态 |
| `longarch/device/callback/{deviceNo}` | **设备 → 后端** | 执行器任务结果回调（taskId 放 payload 里） |
| `longarch/device/command/{deviceNo}` | **后端 → 设备** | 下发控制指令 |

**`{deviceNo}` 必须与数据库 `sensor_device.device_no` 完全一致**，否则后端找不到设备会丢弃消息。

## 三、Payload 格式（JSON，UTF-8）

### 1. 环境4合1传感器
**Topic**: `longarch/device/telemetry/SEN-ENV-DM01`
```json
{
  "temperature": 100,
  "humidity": 100,
  "light": 100,
  "CO2": 100
}
```

### 2. 土壤 NPK 传感器
**Topic**: `longarch/device/telemetry/SEN-NPK-DM01`
```json
{
  "N": 100,
  "P": 100,
  "K": 100
}
```

### 3. 土壤 pH 传感器
**Topic**: `longarch/device/telemetry/SEN-PH-DM01`
```json
{
  "pH": 100
}
```

### 4. 土温土湿传感器
**Topic**: `longarch/device/telemetry/SEN-SM-DM01`
```json
{
  "soilTemp": 100,
  "soilMoisture": 100
}
```

### 字段说明

| 字段 | 单位 | 说明 |
|------|------|------|
| `temperature` | ℃ | 空气温度 |
| `humidity` | % | 空气湿度 |
| `light` | Lux | 光照强度 |
| `CO2` | ppm | 二氧化碳浓度 |
| `N` / `P` / `K` | mg/kg | 氮/磷/钾含量 |
| `pH` | — | 土壤酸碱度 |
| `soilTemp` | ℃ | 土壤温度 |
| `soilMoisture` | % | 土壤湿度 |

> 📌 **`time` / `timestamp` 字段请不要发送**。采样时间统一由服务器在收到消息时自动记录，避免硬件时钟不准导致数据时序错乱。

## 三·附录、网关批量上报（推荐给主控单元使用）

如果一个大棚只有**一个主控单元**统一采集所有传感器，可以把多个传感器的数据打包成一条 MQTT 消息发送，减少报文数量。

### Topic

```
longarch/device/telemetry/batch/{gatewayNo}
```

末段 `{gatewayNo}` 填主控编号即可，例如 `GW-DM01`（1号大棚网关）。

### Payload 总览

| 大棚 | Topic | gatewayNo | 传感器数 | 点位 |
|------|-------|-----------|---------|------|
| 1号 | `longarch/device/telemetry/batch/GW-DM01` | `GW-DM01` | 13 | 1#~5# |
| 2号 | `longarch/device/telemetry/batch/GW-DM02` | `GW-DM02` | 7 | 6#~8# |
| 3号 | `longarch/device/telemetry/batch/GW-DM03` | `GW-DM03` | 13 | 9#~13# |

### 字段说明

| 字段 | 必需 | 说明 |
|------|------|------|
| `gatewayNo` | 可选 | 主控单元编号，仅用于日志，topic 末段也可以 |
| `sensors` | **必需** | 数组，每个元素代表一个传感器 |
| `sensors[].deviceNo` | **必需** | 传感器 deviceNo，必须在后端已注册 |
| `sensors[].data` | **必需** | 该传感器的 payload，字段和第三节单设备格式一致 |

> 📌 **不要发送 `time` / `timestamp` 字段**。采样时间统一由服务器记录，保证全站时序一致。

### 处理逻辑

后端收到后会自动拆开，等价于每个传感器独立上报一次，入库结果和路径 A 完全一致。

### ⚠️ 注意

- 每个 `sensors[]` 元素必须同时有 `deviceNo` 和 `data` 两个键
- 只有一个 sensor 不合法（比如 deviceNo 拼写错误）时，其他 sensor 不受影响，继续入库
- **单设备上报（第三节）和批量上报可以混用**，硬件代码方便哪种就用哪种

## 五、通信规范

1. **上报频率**：建议 30~60 秒一次，紧急告警可随时上报
2. **断线重连**：客户端自动重连，建议指数退避策略（1s → 2s → 4s → … → 60s）
3. **Payload 编码**：UTF-8，纯 JSON
4. **单次消息大小**：建议 < 256 字节
5. **多指标打包**：一个物理传感器一条消息，多个数值字段放同一个 JSON 里。后端会自动拆分入库。
