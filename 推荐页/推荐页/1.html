Longarch Smart Agriculture

智慧农业系统的 核心功能 与
技术栈总览

聚焦业务功能与技术架构全景，面向展示与汇报场景， 清晰呈现系统能力边界、模块分工与工程栈组成。

01
系统定位业务目标
智慧农业认养 + 远程作业
全栈闭环系统
系统围绕“认养关系、地块、设备、任务调度、运营审核、设备回调”构建， 形成覆盖用户、运营与设备层的智慧农业协同闭环。
终端协同 · 管理端 + 小程序端 + 大棚前端
目标链路 · 创建 -> 审核/调度 -> 执行 -> 回写
平台定位于“认养服务 + 智能作业 + 数据回写”一体化， 强调角色边界、任务流转与设备链路协同。
02
角色与权限账号体系
权限边界
明确可控
支持 admin、adopter、operator、guest 四类角色， 前后端路由与接口权限按角色隔离，避免职责混淆。
登录维度 · 管理端账号登录 + 小程序 stub 身份联调
边界收敛 · admin 端已移除 OperatorQueue 入口/路由/API
权限体系强调“谁可见、谁可执行、谁可审核”， 支撑多端协同下的清晰职责分层。
03
认养体系地块管理
认养码/订单/地块
可持续运营
系统支持认养码签发、认养订单管理、地块详情与状态查看， 为后续任务创建与设备联动提供业务基础。
业务实体 · adoption_code / adoption_order / plot
用户路径 · 我的认养 -> 地块详情 -> 申请操作
该模块作为业务入口，承接从用户认养到作业申请的完整路径。
04
任务体系状态机制
任务是系统核心
一切围绕状态流转
pending -> queued/running -> success/failed/cancelled
审核状态 · none / operator_required / approved / rejected
核心表 · operation_task
任务状态与审核状态双轨并行，支持自动调度与人工审核双路径。
05
直派运营新增能力
交由运营处理
可选增强路径
认养者创建任务时可选“交由运营处理”， 任务直接进入 operator_required，保留原自动风险策略并行存在。
风险原因 · manual_direct_operator / manual_reason
指派字段 · assignee_user_id / assignment_mode
该能力让任务处理更灵活，可按场景选择自动策略或人工运营介入。
06
Operator 工作台责任域筛选
待审核任务处理
并支持“仅我的”
operator 工作台支持待审列表、通过/拒绝、责任域全部/仅我的筛选， 并可在联调中验证绑定关系对视图结果的直接影响。
关键能力 · 审核动作 + 视图切换 + 状态刷新
依赖关系 · operator_plot_binding
工作台聚焦待审效率与责任边界，提升运营处理的可控性与可追踪性。
07
责任域配置管理端能力
绑定地块 · 设主责
解绑维护
Operator Scope 页面支持选择操作员、绑定地块责任域、设置主责和解绑， 每次操作均可落库并反馈到 operator 工作台。
页面动作 · 绑定 / 设主责 / 解绑
数据落点 · operator_plot_binding.is_primary/status/deleted
责任域配置是运营组织管理核心，确保任务分配符合地块归属规则。
08
后端技术栈服务能力
Java 17 · Spring Boot · MQTT
业务逻辑稳定承载
任务与设备闭环
MyBatis-Plus + MySQL 负责核心数据模型， Spring Integration + Paho 负责 MQTT 入站/出站消息链路。
核心栈 · Java / Spring Boot / MyBatis-Plus / MySQL
消息链路 · MQTT callback / heartbeat / telemetry
后端负责承载业务规则、数据一致性与设备通信，是平台稳定运行底座。
09
前端技术栈体验能力
管理端 + 小程序端
风格统一演进
管理端基于 React + TS + Vite + Query + GSAP， 小程序端基于 Taro + React + TS + SCSS，并支持 H5 联调。
管理端 · React / TypeScript / Vite / TanStack Query
小程序端 · Taro / React / TypeScript / SCSS
前端体系强调跨端一致的交互体验、状态反馈与信息可读性。
10
测试与工程体系部署架构
Engineering · Testing · Runtime
工程化联调与部署
支撑全链路协同
采用 PowerShell + pnpm + Maven + Mosquitto 进行本地服务编排， 并结合 Python 模拟器与自动化工具支持持续联调与回归测试。
测试方式 · MCP Playwright + Python MQTT 模拟 + 手工冒烟
部署要素 · MQTT Broker / Backend / Admin / Miniapp H5
工程体系覆盖开发、联调、测试与部署关键环节， 保障业务功能可持续演进与稳定交付。
